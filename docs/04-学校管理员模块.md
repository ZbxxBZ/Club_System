# 学校管理员模块

## 一、模块概述

学校管理员（SCHOOL_ADMIN）负责全局管理，包括用户管理、社团审批与管理、活动审批、经费审批、数据统计和系统配置。所有接口以 `/api/school-admin/` 为前缀。

## 三、核心审批流程

### 3.1 社团新建审批（3 阶段流程）

```
学生提交申请 → apply_status=1 (待初审/材料审核)
    ↓ 学校管理员操作
status 1 → 2 (答辩审核)
    ↓
status 2 → 3 (公示审核)
    ↓
status 3 → 4 (通过) ← 终态
    或
status 1/2/3 → 5 (驳回) ← 终态
```

**通过时的联动操作**:
1. 更新 club.status → 2（活跃）
2. 设置 club.established_date 为当天
3. 将申请发起人的角色从 STUDENT 升级为 CLUB_ADMIN，绑定 club_id
4. 清除发起人的会话缓存

**可配置审批阶段**:
- Redis Key: `config:approval:club-stages`
- 格式: `[{"from":1,"to":2},{"from":2,"to":3},{"from":3,"to":4}]`
- 未配置时使用硬编码的 3 阶段流程
- 驳回(→5) 始终允许从任何非终态发起

### 3.2 社团注销审批

```
社团管理员提交注销 → cancel_status=1 (待公示)
    ↓
status 1 → 2 (待经费结清)
    ↓
status 2 → 3 (待资产移交)
    ↓
status 3 → 4 (已完成) ← 终态
    或
status 1/2/3 → 5 (驳回) ← 终态
```

**通过时(status=4)联动操作**:
1. 更新 club.status → 5（已注销）
2. 将社团发起人角色降级为 STUDENT
3. 清除发起人会话缓存

**驳回时(status=5)联动操作**:
1. 更新 club.status → 2（恢复活跃）

### 3.3 活动审批

```
社团管理员创建活动 → event_status=2 (待审核)
    ↓ 学校管理员操作
APPROVE → event_status=3 (报名中)
  + 初始化 Redis 名额配额: event:quota:{eventId} = limitCount - currentSignups
    ↓
REJECT → event_status=5 (已驳回, 记录 reject_reason)
  社团管理员可修改后重新提交
```

### 3.4 经费审批

```
社团管理员提交 >500元 支出 → expense_status=1, approve_level=2
    ↓ 学校管理员操作
APPROVE → expense_status=2
  + 写入台账: balance = prevBalance - amount
    ↓
REJECT → expense_status=3 (记录 reject_reason)
  社团管理员可修改后重新提交
```

## 四、用户与社团管理

### 4.1 修改用户状态 (`updateUserStatus`)
- 仅允许 ACTIVE(1) ↔ FROZEN(2) 互转
- 修改后清除 Redis 会话缓存，强制下次请求重新鉴权

### 4.2 修改用户角色 (`updateUserRole`)
- 更新 user_role 记录的 role_id 和 club_id
- 禁用该用户其他活跃角色记录
- 清除会话缓存

### 4.3 社团管理
- **编辑限制**: status=5（已注销）或 apply_status IN (1,2,3)（审核中）的社团不可编辑
- **冻结/解冻**: 仅 status=2（活跃）和 status=3（冻结）之间可互转
- 所有操作写入审计日志（包含操作前后快照）

### 4.4 毕业生处理 (`autoExitGraduates`)
- 查找 graduate_at <= NOW() 的用户
- 将其从所有社团成员中退出（member_status=2, quit_at=NOW()）

### 4.5 冻结注销社团账号 (`autoFreezeCanceledClubAdmins`)
- 查找 club.status=5（已注销）的社团管理员
- 将对应用户 status 改为 2（冻结）

## 五、数据统计

### 5.1 概览统计 (`statisticsOverview`)
- 活跃社团数 / 社团总数
- 活跃用户数 / 用户总数
- 待审批数量
- 异常支出数量

### 5.2 社团统计 (`getClubStatistics`)
- 分类分布（体育/科技/文艺/文化/公益等）
- 状态分布
- 年度新建/注销数量
- 学生覆盖率（加入社团的学生比例）
- TOP10 活跃社团
- 成员规模分布（0-10, 11-30, 31-50, 50+）

### 5.3 活动统计 (`getEventStatistics`)
- 年度活动总数 / 总报名人次
- 月度活动趋势
- TOP10 活动社团
- 活动状态分布

### 5.4 财务统计 (`getFinanceStatistics`)
- 学校拨款总额 / 总收入 / 总支出 / 总余额
- 收入类型分布（拨款/赞助/会费）
- 各社团财务汇总
- 月度收支趋势

### 5.5 财务审计报告 (`getFinanceAuditReport`)
- 指定社团的年度报告
- 月度收支明细（1-12月）
- 分类支出统计
- 异常支出数量

## 六、系统配置管理

### 经费审批阈值
- Redis Key: `config:approval:expense-threshold`
- 默认值: "500"
- 校验: 非空、合法数字、非负
- 影响: ClubAdminPermissionServiceImpl 中 createExpense/resubmitExpense 的审批级别判断

### 社团审批阶段
- Redis Key: `config:approval:club-stages`
- 默认值: `[{"from":1,"to":2},{"from":2,"to":3},{"from":3,"to":4}]`
- 校验: 合法 JSON 格式 `List<Map<String, Integer>>`
- 影响: validateApprovalTransition 的状态流转校验

## 七、审计日志

所有关键操作记录到 `audit_log` 表:
- `biz_type`: CLUB_APPLY, CLUB_CANCEL, CLUB_STATUS, CLUB_MANAGE, CLUB_JOIN_APPLY, EVENT, EXPENSE, INCOME
- `op_type`: APPROVE, REJECT, CREATE, UPDATE, DELETE, FREEZE, UNFREEZE, RESUBMIT
- 记录: 操作人信息、IP、User-Agent、操作前后 JSON 快照、备注
