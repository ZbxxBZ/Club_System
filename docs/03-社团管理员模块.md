# 社团管理员模块

## 一、模块概述

社团管理员（CLUB_ADMIN）管理自己所属社团的全部事务，包括社团信息、组织架构（职位+成员）、入社审批、活动管理、经费管理。所有接口均以 `/api/club-admin/me/` 为前缀，通过 `resolveMyClubId()` 自动解析当前管理员管理的社团。

## 三、核心功能逻辑

### 3.1 社团信息管理

**获取社团信息** (`getMyClub`):
- 通过 `resolveMyClubId(user)` 从当前用户的 user_role 记录找到绑定的 club_id
- 查询 club 表返回: clubName, category, introduction(映射自description), purpose, instructorName

**修改社团信息** (`updateMyClub`):
- 校验至少有一个字段非空
- 若修改社团名称，检查是否重名
- 更新 club 表对应字段

**招新配置** (`updateMyClubRecruitConfig`):
- 校验 recruitStatus 为 OPEN 或 CLOSED
- 若 OPEN: recruitStartAt, recruitEndAt, recruitLimit 均必填
- recruitLimit 范围 1-10000
- recruitEndAt 必须晚于 recruitStartAt

### 3.2 职位管理（树形结构）

**创建职位** (`createMyClubPosition`):
```
校验: positionName 非空, levelNo >= 1
    ↓
若指定 parentPositionId → 校验父职位存在且属于本社团
    ↓
计算 levelNo: 有父级 → 父级 level + 1; 无父级 → 1
    ↓
计算 sortNo: 同父级下最大 sortNo + 10
    ↓
插入 club_position 记录
```

**修改职位** (`updateMyClubPosition`):
```
校验职位存在且属于本社团
    ↓
若修改 parentPositionId → 防止循环引用 (ensureNoCycle)
    ↓
更新 positionName, parentPositionId, levelNo, sortNo
```

**删除职位** (`deleteMyClubPosition`):
```
校验职位存在且属于本社团
    ↓
检查是否有子职位 → 有则拒绝
    ↓
检查是否有成员占用此职位 → 有则拒绝
    ↓
软删除 (is_deleted = 1)
```

**循环引用检测** (`ensureNoCycle`):
- 从候选父级向上遍历父链，如果遇到当前职位 ID 则存在循环，抛出异常

### 3.3 入社审批 (`decideJoinApply`)

```
查询申请记录(带 FOR UPDATE 行锁)
    ↓
校验: 申请属于本社团 且 join_status=1(待审批)
    ↓
APPROVE:
  - 更新 join_status=2, 记录 review_user_id 和 reviewed_at
  - 插入 club_member 记录(member_status=1)
  - 若指定了职位(positionId/positionName) → 关联职位
  - 写入审计日志
    ↓
REJECT:
  - 更新 join_status=3, 记录 reject_reason
  - 写入审计日志
```

### 3.4 活动管理

**创建活动** (`createMyEvent`):
```
校验: title, location, startAt, endAt 必填
  startAt 必须在未来, endAt > startAt
  limitCount >= 0
    ↓
生成签到码(6位随机字母数字)
    ↓
插入 event 记录(event_status=2 待审核)
    ↓
返回活动详情
```

**重新提交** (`resubmitMyEvent`):
- 仅允许 event_status=5(已驳回) 的活动重新提交
- 更新活动信息，状态改回 2(待审核)，清除驳回原因

**管理员签到** (`checkinMyEvent`):
```
校验: 活动属于本社团, 状态为 3 或 6
    ↓
校验: 目标用户已报名且未签到
    ↓
尝试恢复软删除的签到记录 → 失败则新建
```

**活动总结** (`submitMyEventSummary`):
- 将 `summaryImages` (List<String>) 序列化为 JSON 字符串存储
- 支持新增和更新（已有总结则更新）

### 3.5 经费管理

**收入录入** (`createIncome`):
```
校验: incomeType(1学校拨款/2赞助/3会费), amount > 0, occurAt, proofUrl 必填
    ↓
插入 income 记录
    ↓
写入台账: 获取最新余额 → 新余额 = 旧余额 + amount → 插入 ledger
```

**支出申请** (`createExpense`):
```
校验: amount > 0
    ↓
判断审批级别:
  amount <= 阈值(默认500, 可配置) → approveLevel=1 (社团自审, 自动通过)
  amount > 阈值 → approveLevel=2 (需学校审核)
    ↓
插入 expense 记录
    ↓
若自动通过(approveLevel=1):
  写入台账: 余额 = 旧余额 - amount → 插入 ledger
```

**审批阈值**:
- 从 Redis 读取 `config:approval:expense-threshold`
- 未配置则使用默认值 500 元
- 学校管理员可通过配置接口动态修改

**支出重新提交** (`resubmitExpense`):
- 仅允许 expense_status=3(已驳回) 的支出重新提交
- 重新计算审批级别，更新支出信息

**余额查询** (`getMyBalance`):
```
balance = findLatestClubBalance() (最后一条台账的 balance_after)
pendingExpense = sumPendingExpensesByClub() (status=1 的待审支出总额)
availableBalance = balance - pendingExpense
```

### 3.6 台账明细 (`listMyLedger`)

- 支持按 bizType(1收入/2支出) 筛选
- 支持按时间范围 startTime~endTime 筛选
- 分页返回，按 occur_at DESC 排序
- 每条记录: 变动金额 + 变动后余额

### 3.7 社团注销 (`submitMyClubCancel`)

```
校验: applyType(1主动/2强制), applyReason, assetSettlementUrl 必填
    ↓
检查是否已有进行中的注销申请 → 有则拒绝
    ↓
更新社团状态为 4(待注销)
    ↓
插入 club_cancel 记录(cancel_status=1)
```

## 四、辅助方法

| 方法 | 说明 |
|------|------|
| `resolveMyClubId(user)` | 从 user_role 查找 CLUB_ADMIN 角色绑定的 club_id |
| `resolvePositionId(clubId, positionId, positionName)` | 按 ID 或名称解析职位 |
| `calculateLevelNo(parentPosition)` | 父级 level + 1，无父级则 1 |
| `ensureNoCycle(positionId, newParentId)` | 遍历父链检测循环引用 |
| `nextSortNo(clubId, parentId)` | 同级最大 sortNo + 10 |
| `getAutoApproveThreshold()` | 从 Redis 读取经费自动审批阈值，默认 500 |
