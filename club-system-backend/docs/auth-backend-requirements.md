# 登录与注册后端对接说明

## 1. 文档目标
本说明用于前后端联调，明确学校社团管理系统在“登录/注册”场景下的接口规范、业务逻辑、状态码与安全要求。

---

## 2. 接口总览
前端当前约定的接口如下：

- 登录：`POST /api/auth/login`
- 注册：`POST /api/auth/register`

前端基础地址可通过 `.env` 配置：

```env
VITE_API_BASE_URL=http://localhost:8080/api
```

---

## 3. 统一返回格式（建议）
建议后端统一返回以下结构，便于前端统一处理：

```json
{
  "code": 0,
  "message": "success",
  "data": {}
}
```

说明：
- `code = 0`：业务成功
- `code != 0`：业务失败（如账号不存在、密码错误、账号被冻结）
- `message`：给前端展示的可读提示
- `data`：业务数据

HTTP 状态码建议：
- `200`：请求成功（业务成功或业务失败都可通过 `code` 区分）
- `400`：参数错误
- `401`：未认证/登录失效
- `403`：无权限
- `409`：注册冲突（账号已存在）
- `500`：服务端异常

## 4. 登录接口规范
### 4.1 请求
`POST /api/auth/login`

Request Body：

```json
{
  "username": "20230001",
  "password": "123456"
}
```

### 4.2 成功响应示例

```json
{
  "code": 0,
  "message": "登录成功",
  "data": {
    "token": "jwt-token",
    "userId": 1001,
    "username": "20230001",
    "realName": "张三",
    "permissionType": "STUDENT"
  }
}
```

> 前端当前会读取 `data.token`（兼容 `data.accessToken`），并根据 `permissionType`（兼容 `role/roleCode/userRole/roles[0]`）执行角色路由跳转。
>
> 权限类型建议值：
> - `STUDENT` -> `/dashboard/student`
> - `CLUB_ADMIN` -> `/dashboard/club-admin`
> - `SCHOOL_ADMIN` -> `/dashboard/school-admin`

### 4.3 后端登录逻辑必须完成
1. 参数校验：`username`、`password` 非空，长度与格式合法。
2. 用户查询：按账号查用户（支持学号/工号）。
3. 状态校验：账号是否启用、是否被冻结、是否已注销。
4. 密码校验：加盐哈希比对（如 BCrypt/SM3+盐）。
5. 登录风控（建议）：失败次数限制、异常 IP 预警、验证码策略。
6. 令牌签发：生成 JWT（或会话令牌），携带用户 ID、角色、过期时间。
7. 会话存储（建议）：将 token/session 写入 Redis，支持主动失效。
8. 审计日志：记录登录结果、IP、UA、时间。
9. 返回用户基础信息与角色：供前端路由与权限控制。

## 5. 注册接口规范
### 5.1 请求
`POST /api/auth/register`

Request Body：

```json
{
  "username": "20230002",
  "password": "123456"
}
```

> 如你后端要求二次确认密码，可扩展 `confirmPassword` 字段。

### 5.2 成功响应示例

```json
{
  "code": 0,
  "message": "注册成功",
  "data": {
    "userId": 1002
  }
}
```

### 5.3 后端注册逻辑必须完成
1. 参数校验：账号格式（学号/工号规则）、密码复杂度（长度、字符类型）。
2. 唯一性校验：账号是否已存在（数据库唯一索引 + 业务校验双保险）。
3. 身份来源校验（推荐）：与校园统一身份认证系统核验账号真实性。
4. 初始化信息：默认角色（如 `STUDENT`）、默认状态（启用）、创建时间。
5. 密码安全存储：强哈希（如 BCrypt），严禁明文。
6. 事务提交：用户基础信息、角色关系等数据原子写入。
7. 注册审计日志：记录注册来源、IP、时间。
8. 可选：注册成功后发送通知（站内信/短信）。

---

## 6. 错误码建议
| code | 含义 | 前端提示建议 |
|---|---|---|
| 0 | 成功 | - |
| 1001 | 参数不合法 | 输入信息有误 |
| 1002 | 账号不存在 | 账号或密码错误 |
| 1003 | 密码错误 | 账号或密码错误 |
| 1004 | 账号已禁用/冻结 | 账号不可用，请联系管理员 |
| 1005 | 账号已存在 | 该账号已注册 |
| 1006 | 密码强度不足 | 密码至少8位，含字母和数字 |
| 1007 | 登录已失效 | 请重新登录 |
| 1099 | 系统异常 | 系统繁忙，请稍后再试 |

---

## 7. 安全要求（必须）
1. 全链路 HTTPS（校内环境同样建议开启）。
2. 密码不可逆存储，禁止日志打印明文密码。
3. 防暴力破解：登录失败次数限制 + 临时锁定策略。
4. Token 过期与刷新策略（建议接入 refresh token）。
5. 关键接口加限流（网关或应用层）。
6. 敏感操作记录审计日志，满足学校合规要求。
