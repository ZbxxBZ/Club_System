# 对话存档

## 1. 项目目标与背景（已确认）
- 项目：基于国产中间件的学校社团管理系统。
- 目标：覆盖社团全生命周期（成立/年审/注销）、用户与权限、活动管理、经费管理、统计报表。
- 重点问题：高并发报名超卖、审批流程慢、状态同步不及时、经费监管与可追溯性不足。

## 2. 系统设计（已确认）
- 前端：Vue3 + Vite + Element Plus，Axios，WebSocket。
- 后端：Spring Boot + MyBatis + MySQL。
- 架构分层：Controller -> Service -> Mapper。
- 关键中间件规划（需求层面）：
  - 应用服务器：TongWeb
  - 缓存：TongRDS
  - 负载均衡：TongHttpServer
  - 消息队列：TongLINK/Q
- 数据层：MySQL 主从、读写分离、分库分表（设计目标）。

## 3. 当前代码实现范围（本次对话已落地）

### 3.1 用户与权限模块核心能力
- 角色体系：`STUDENT` / `CLUB_ADMIN` / `SCHOOL_ADMIN`。
- 鉴权控制：学校管理员接口需要 `SCHOOL_ADMIN` 才可调用。
- 用户状态控制：支持 `ACTIVE` 与 `FROZEN` 的双向切换。
- 角色切换：支持学生与社团管理员双向切换（`STUDENT <-> CLUB_ADMIN`），并支持设置为 `SCHOOL_ADMIN`（按接口约束）。

### 3.2 “设为社团管理员”接口（已按文档对齐）
- 接口：`PATCH /api/school-admin/users/{userId}/role`
- 已实现：
  - 入参校验：`roleCode` 仅允许 `STUDENT|CLUB_ADMIN|SCHOOL_ADMIN`。
  - 事务处理：角色变更在事务内执行。
  - 用户存在性校验：目标用户不存在时返回明确业务错误。
  - 生效策略：当前实现为“覆盖生效角色”（先停用当前有效角色，再激活/插入目标角色）。
  - 返回数据：`userId`、`roleCode`、`effectiveRoleCode`。
  - 控制器提示语：
    - 设为管理员：`角色调整成功`
    - 取消管理员：`已取消社团管理员`
- 列表联动修复：
  - 用户列表角色取值改为取最新有效角色（按 `updated_at desc, id desc`），避免“接口成功但列表仍显示学生”。

### 3.3 “禁用/取消禁用账号”接口（已按文档对齐）
- 接口：`PATCH /api/school-admin/users/{userId}/status`
- 已实现：
  - 入参校验：`status` 仅允许 `ACTIVE|FROZEN`。
  - 事务处理：状态更新加入事务。
  - 合法性校验：仅允许在可切换状态内执行（当前实现仅对 `ACTIVE/FROZEN` 双向切换）。
  - 幂等处理：重复提交同状态不报错，直接成功返回当前状态。
  - 控制器提示语：
    - 禁用成功：`账号已禁用`
    - 取消禁用成功：`已取消账号禁用`
  - 返回数据：`userId`、`status`。

## 4. 重要约束与决策（已统一）
- 非审批类功能（用户状态切换、角色切换、定时任务启停等）：
  - **不写 `audit_log`**
  - 也按你的要求：**不写其他操作日志表**
- `audit_log` 仅用于社团审批业务链路（如 `APPROVE/REJECT/RETURN`）。
